# autoeval — Autonomous Optimization Loop Scaffolder

A Claude Code skill that transforms a vague optimization problem into a fully scaffolded, runnable autonomous experiment loop — bridging the gap between "I have an idea" and "I have an autonomous experiment running overnight."

## What it does

The `/autoeval` command guides you through defining your optimization problem, designing an automatic scoring function, and building a complete eval suite — then scaffolds everything the meta-agent needs to run overnight: a `program.md` directive, seed implementation, monitoring dashboard, and loop runner.

### Key features

- **Problem classification** — maps your problem against a 12-type loop taxonomy (Training, Agent Harness, Generative Output, Algorithm Performance, and 8 more)
- **Metric design** — interactive exploration of scoring functions with stress-testing for gameability, signal strength, and cost
- **Eval suite building** — the crown jewel. Coverage strategy, test cases, scoring functions, integrity validation, and user review
- **Eval integrity validation** — catches evals that "grade their own homework" (synthetic references generated by the same technique as the system under test)
- **Dual-model architecture** — a runner model (sonnet) handles iteration while a deep reasoning model (opus) is called for hypothesis generation and exploration rounds
- **Live monitoring dashboard** — `monitor.py` serves a real-time chart showing score progression, component scores, and experiment history
- **Steering** — guide the loop agent mid-run via `steering.md` without stopping the loop
- **Auto-restart loop runner** — `run-loop.py` manages session restarts to prevent context window exhaustion
- **Edit surface marking** — clearly separates what the meta-agent can modify from fixed infrastructure
- **Exit ramp** — identifies problems that aren't suited for optimization loops and says so, rather than building a bad loop
- **Socrates integration** — dialectic stress-testing at key decision points (optional, via [zetaminusone/socrates](https://github.com/ZetaMinusOne/socrates))

## Usage

```
/autoeval <describe your optimization problem>
/autoeval --auto <describe your optimization problem>
```

**Examples:**

```
/autoeval optimize my sorting algorithm for speed on large datasets
/autoeval improve my classifier's accuracy on the test suite
/autoeval --auto tune my prompt to maximize extraction accuracy
```

The `--auto` flag makes Phases 3-6 run autonomously after the interactive metric design is locked in.

## How it works

autoeval runs through 6 phases:

1. **Problem Definition** (interactive) — clarify what you're building, classify the loop type, exit if it's not an optimization problem
2. **Metric Design** (interactive) — find an automatic scoring function, stress-test it for gameability and signal strength
3. **Eval Suite** (interactive) — build test cases, scoring functions, integrity validation, and user review
4. **Harness Scaffolding** — generate a minimal seed implementation with marked edit surface
5. **Program.md Generation** — write the meta-agent directive with dual-model iteration protocol and constraints
6. **Loop Setup** — wire everything together, verify baseline, provide kickoff command

Phases 1-2 are always interactive. Phases 3-6 are interactive by default, or autonomous with `--auto`.

### What it produces

| Artifact | Description |
|----------|-------------|
| `program.md` | Meta-agent directive — goal, edit surface, eval commands, keep/discard rules, session limits |
| Seed harness | Minimal baseline implementation with clearly marked edit surface |
| `evals/` | Test cases, scoring functions, runner script, coverage documentation |
| `run-loop.py` | Loop runner — launches sessions with auto-restart and hard timeout |
| `monitor.py` | Live dashboard — score progression chart on localhost:8080 |
| `steering.md` | Steering file — guide the agent mid-run (see Steering below) |
| Environment files | Dependencies, Dockerfile (optional), .gitignore |

### Running the loop

After autoeval completes, you get two commands:

**Terminal 1 — run the loop:**

```bash
cd <output-dir>
python run-loop.py
```

This launches claude sessions that auto-restart every N iterations with fresh context. A hard timeout acts as a safety net. Ctrl+C once kills the current session; Ctrl+C again stops the loop.

**Terminal 2 — monitor progress:**

```bash
cd <output-dir>
python monitor.py
```

Opens a live dashboard at `http://localhost:8080` showing composite score over iterations, individual component scores (toggleable), best score line, hypothesis text on hover, and summary stats.

### Dual-model architecture

The loop uses two models to balance cost and quality:

- **Runner model** (default: sonnet) — handles the main iteration cycle: editing code, running evals, deciding keep/revert, committing
- **Deep reasoning model** (default: opus) — called via `claude -p` one-shot for hypothesis generation and exploration rounds

This gives opus-level creativity for the "think hard" steps at ~10% of the cost of running opus for the entire loop.

## Steering

Steer the loop agent mid-run without stopping it. Append entries to `steering.md` tagged with the commit hash they apply after:

```markdown
## after 7f5ad6f
Stop trying to optimize the greedy heuristic. The biggest gain is switching 
to a scoring-based approach where you evaluate all valid bins and pick the 
best one. Look at the best-fit-decreasing literature.

## after a1b2c3d
The classifier accuracy is plateauing at 96.2%. The bottleneck is feature 
engineering, not model choice. Try polynomial features on columns 3-7 — 
those have non-linear relationships with the target.
```

**How it works:**

1. Before each iteration, the loop agent checks `steering.md`
2. It finds the entry whose commit hash is closest to the current HEAD
3. That entry becomes active guidance — it overrides the agent's own hypothesis when they conflict
4. The steering context is also passed to the deep reasoning model during hypothesis generation

**Two levels of control:**

| File | Purpose | Takes effect |
|------|---------|-------------|
| `steering.md` | Per-iteration guidance (ephemeral) | Next iteration |
| `program.md` | Permanent directives and constraints | Next session restart |

Steering entries accumulate — old ones are automatically irrelevant once HEAD moves past them. The full steering history is preserved for reference.

## Eval integrity

Phase 3 includes an integrity validation step that catches evals that look functional but measure nothing real:

- **Self-referential references** — synthetic references generated by the same technique as the system under test ("grading your own homework")
- **Discriminating power** — can the eval tell good output from bad? Tested by comparing the seed implementation against a deliberately degraded version
- **Ceiling reachability** — is the theoretical best score actually achievable, or is the scoring function structurally capped?

The user also reviews the eval in plain language before proceeding: what's being scored, what's NOT being scored, what different score levels mean concretely, and how the agent could game the eval.

## Loop type taxonomy

autoeval classifies problems against 12 loop types:

| # | Loop Type | Edit Surface | Scoring Function |
|---|-----------|-------------|-----------------|
| 1 | Training | Model/training code | Validation loss, accuracy |
| 2 | Agent Harness | Prompt, tools, orchestration | Task pass rate, benchmark |
| 3 | Generative Output | Synthesis algorithms | Reference similarity, LLM-as-judge |
| 4 | Algorithm Performance | Implementation code | Speed, memory, accuracy |
| 5 | Retrieval & Ranking | Chunking, indexing, re-ranking | Recall, precision, MRR |
| 6 | Pipeline Optimization | Processing stages, filters | Throughput, quality, cost |
| 7 | Simulation Calibration | Sim parameters, physics models | Fit to real-world data |
| 8 | Strategy & Decision | Decision rules, heuristics | Win rate, expected value |
| 9 | Adversarial | Attack/defense mechanisms | Vulnerability rate, resistance |
| 10 | Data Curation | Dataset composition, filtering | Downstream model performance |
| 11 | Control Systems | Controller params, architecture | Tracking error, stability |
| 12 | Interface Optimization | UI layout, interaction flows | Task completion, engagement |
